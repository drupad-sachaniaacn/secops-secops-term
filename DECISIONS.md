# Decisions Log

Running record of design decisions made during the build of SecOps Terminal. Every non-obvious choice gets an entry here, in chronological order. Format:

> **Decision** — short context — implication / rationale.

## Locked by brief v3 §13 (do not relitigate)

These are reproduced here so the rationale is in-tree, not only in the brief.

1. **Vision One region: US, base URL hardcoded.** `https://api.xdr.trendmicro.com`. Not configurable post-decision. Removes a config knob and a class of misconfiguration bugs for the single-tenant deployment.
2. **Deep Security scope v1: read-only.** Alerts list and agent status only. No policy reads, no writes. Smaller surface for v1 means fewer auth-scope mistakes and a clean "phase write features later" boundary.
3. **Webhook notifiers: pluggable registry under `secops_term/notifications/`.** Built-ins: `generic_json`, `slack`, `teams`. Slack/Teams webhook URLs are credentials and live in keyring. Multi-instance via `{notifier}:{instance}`.
4. **Threat intel providers: pluggable registry under `secops_term/intel/providers/`.** Phase 1 minimum: `abuse_ch`, `otx`, `rss`. Phase 6: `virustotal`, `greynoise`, `abuseipdb`, `nvd`. Multi-instance supported. Each provider's API token stored in keyring.
5. **Health checks across the board.** Every provider and every notifier implements `async health_check() -> HealthStatus`. CLI: `config test <provider>` and `config test-all`. Wizard auto-probes on save. Doctor calls `test-all`. VT probe uses `/users/<owner>` to avoid burning the daily quota.
6. **Keyring fallback: Argon2id-Fernet with session passphrase, default-on.** Argon2id parameters: `memory_cost=65536`, `time_cost=3`, `parallelism=4`, salt 16 bytes. `--strict-keyring` and `keyring.strict=true` refuse fallback.
7. **Audit log: hash-chained JSONL.** `sha256(prev_hash || canonical_json(entry))`. Genesis prev_hash is 64 zeros. `secops-term audit verify` walks the chain across rotations. Probe entries marked `kind="probe"` for filtering.
8. **No environment variables.** Not used for secrets and not used for environment-specific URLs. Eliminates leakage paths via shell history, process listings, CI logs, container inspect, and accidental commits.
9. **TLS always verified.** No `--insecure` flag exists.
10. **HTTP scheme: `https` only externally.** `http` allowed only on loopback (MCP) and per-source-opt-in for RSS feeds (logged on use).
11. **Playbook templating: custom AST-walking sandbox, not Jinja2.** Whitelist of AST node types only. No `Call`, no `Lambda`, no dunder access. Test corpus of injection attempts must all fail validation.
12. **Subprocess: `create_subprocess_exec` only.** Never `_shell`, never `shell=True`. The `claude` binary path is resolved once at startup via `shutil.which` and pinned for the session.
13. **MCP server: loopback only.** Optional shared-secret auth via `Authorization: Bearer <secret>`.
14. **AI on untrusted content.** Sentinel-fenced with random IDs. Output is advisory display only — never drives playbook control flow (no `when:`, no write-step parameters).
15. **Chronicle auth.** Service account JSON only. Content moved to keyring on configuration; source file optionally shredded with confirmation.

## Implementation choices made during scaffolding

- **Build backend: `hatchling`.** PEP 517 standard, minimal config, plays well with `uv`. Alternative was `setuptools` — rejected for the extra `MANIFEST.in` baggage.
- **Package name: `secops_term` (Python module), `secops-term` (PEP 621 distribution name and CLI entry point).**
- **Ruff lint set: `E,F,I,B,S,UP,N,W,ASYNC,RUF`.** `S` (bandit) is the security-relevant set; `S608` enforced — hardcoded SQL is a CI failure. Tests get per-file ignores for `S101/S105/S106` (asserts and password false positives).
- **Mypy: `strict = true`.** Plus `warn_unreachable`, `warn_redundant_casts`, `warn_unused_ignores`. Untyped third-party deps listed in an `ignore_missing_imports` overrides block.
- **CI: GitHub Actions.** Two jobs — `qa` (ruff + mypy + non-security pytest) and `security` (pytest -m security). If the project moves to Azure DevOps / GitLab, replace `.github/workflows/ci.yml` with the equivalent. Logged here so the choice is visible.
- **Pytest markers:** `security`, `slow`, `network`. `network` tests are skipped by default; opt in with `-m network`. Strict markers and strict config so a typo'd marker fails the run.
- **Coverage source: `secops_term`, branch coverage on.** Excludes the standard `if __name__ == "__main__":` and `if TYPE_CHECKING:` blocks.
- **Empty `requirements.lock` placeholder.** Populated by `uv pip compile pyproject.toml -o requirements.lock` — left empty in the scaffold so the file is at the canonical path before any real lock is generated.
- **`.gitignore` blocks `.env*` even though we never read env vars.** Defense in depth — a stray `.env` should never reach the repo regardless.

## Chunk 2 — foundational core (paths, redact, secrets, audit, db)

- **`paths` root override.** `paths.set_root_for_tests(...)` is the only sanctioned way to point the user-data root somewhere other than `~/.secops-term/`. Production code never calls it. The `tmp_root` pytest fixture uses it.
- **POSIX mode bits enforced strictly.** `verify_restrictive_acl` rejects any file with `mode & 0o077 != 0` (i.e., any group/world bit set). Files: `0o600`. Directories: `0o700`. Apply on every write, verify on every read.
- **Windows ACL via `icacls`, verify via `pywin32`.** `apply` shells out to `icacls /inheritance:r /grant:r <user>:F SYSTEM:F`. `verify` enumerates the DACL via `win32security.GetFileSecurity` and rejects if any principal beyond the current user and SYSTEM appears. Cross-platform `# type: ignore[import-not-found,import-untyped,unused-ignore]` on the win32security import so mypy passes on both POSIX and Windows.
- **Tainted-string registry has a 4-char minimum.** Refusing to taint values shorter than 4 chars avoids redacting unrelated substrings in arbitrary text. Empty values are silently ignored. `__repr__` exposes only a count, never contents.
- **Argon2id-Fernet file format** `~/.secops-term/secrets.enc`: 27-byte binary header (`>4sBIBB16s` = magic `STSE`, version `1`, mem_cost `65536`, time_cost `3`, parallelism `4`, salt 16 bytes) followed by the urlsafe-b64 Fernet token. JSON dict `{"<provider>:<instance>:<field>": "<value>"}` is the encrypted payload.
- **Salt is rotated only on full passphrase rotation.** Within a session, a single salt is reused for re-encryption on every `set_secret`/`delete_secret`. Argon2id key derivation runs once per session — derived key cached in a `bytearray`, zeroized on `atexit` and on SIGINT/SIGTERM.
- **`get_secret` automatically taints the returned value** with label `<provider>:<instance>:<field>`. Callers can use the value freely; any leak into logs or tracebacks is redacted via the global registry.
- **Keyring backend detection.** Heuristic: `type(backend).__name__.lower()` containing `"fail"` or `"null"` → not working. Avoids probing real keyrings with side-effect writes. Tests inject a fake keyring module via `SecretsConfig.keyring_module`.
- **Migration runner uses `execute()` per-statement, not `executescript()`.** Python's sqlite3 `executescript()` issues an implicit `COMMIT` first, which would close any wrapping `BEGIN`. Splitting on `;` and running each statement via `execute()` lets us wrap the whole migration in a manual `BEGIN`/`COMMIT` for atomicity. Migration files are author-controlled and must not embed `;` in string literals.
- **`isolation_level=None` (autocommit) plus explicit BEGIN/COMMIT.** This combination works on Python 3.11+ and gives us atomic migrations without relying on the 3.12+ `autocommit=False` parameter.
- **Audit rotation filenames use microsecond precision** (`audit-%Y%m%d-%H%M%S-%f.jsonl`). Second-precision collides under fast rotation pressure (e.g. small `rotation_size` in tests) and `os.replace` would silently overwrite earlier rotations.
- **Audit redaction at emit, hash over redacted form.** Entries pass through `redact.redact()` recursively (str leaves only) before hashing, so the chain is portable: an external auditor can verify the chain without ever seeing the original secrets.
- **`N818` disabled.** Pep8-naming would force every exception to end with `Error`. Our exceptions (`ChainBroken`, `KeyringUnavailable`, `StrictKeyringViolation`, `PassphraseRequired`, `CorruptSecretsFile`) read better with descriptive names. Stylistic preference logged here so it's not relitigated.
- **`DBError` raised in place of `assert` for type narrowing** in `_init_meta_unlocked` — avoids ruff `S101` and gives a real runtime error if the invariant is ever violated under `python -O`.
- **No concrete migrations shipped in Phase 0.** `secops_term/core/migrations/` is an empty package. Tests cover the runner with synthetic `Migration` objects. Real migrations land in Phase 1+ as feature modules need them.

## Chunk 3 — network core (http, url_guard, health)

- **`validate_url` returns `ValidatedURL` with a pinned IP, but transport-level pinning is deferred to Phase 2.** The brief mandates "pin the resolved IP, set Host header to the original hostname for SNI/TLS, implement via httpx transport." Doing this correctly requires a custom `httpx.AsyncBaseTransport` that connects to a literal IP while passing the hostname for SNI/cert verification — non-trivial and only needed once we wire up real provider clients. Phase 0 ships the validation primitive (`ValidatedURL.pinned_ip`, `host_header`, `url_with_pinned_ip`) so callers can use it; the transport wiring lands when Chronicle/V1 clients arrive in Phases 2-3.
- **CGNAT (100.64.0.0/10) explicitly rejected** even though Python 3.13+ flags it via `is_private`. Defense in depth — keeps the check correct on older interpreters and makes the intent visible.
- **Any-private-rejects-all DNS rebinding mitigation.** If `getaddrinfo` returns multiple addresses and ANY is private, the URL is rejected. This catches DNS round-robin poisoning where the attacker registers a public + a private A record.
- **Per-request retry budget (not per-host sliding window).** The brief uses "per-host" framing but the underlying need is "don't hammer a flaky endpoint forever". 3 attempts per request with exponential backoff achieves that for Phase 0. A sliding-window per-host budget can be added in a later phase if quota burn becomes an issue.
- **Backoff jitter via `secrets.randbelow`.** Cryptographic randomness is overkill for jitter, but it costs nothing here and keeps ruff `S311` satisfied without `# noqa` clutter.
- **HTTPConfig hard ceiling = 200 MB.** Checked at `HardenedClient.__init__` (config-level) AND in `request()` (per-call override). Both paths raise `CapTooHigh`. Failing fast at construction surfaces misconfiguration before any network call.
- **Streaming + chunk-counting for response cap.** Handles servers that omit `Content-Length` (chunked transfer encoding) and servers that lie about it. The cap is enforced both ways: header-based when present, byte-counting always.
- **`HardenedClient` exits the retry loop with the last response when budget is exhausted on retryable status.** A different design would raise `RetryBudgetExhausted` at the end, but returning the last 5xx lets callers inspect the response (e.g., for `Retry-After` headers) and decide their own policy. Transport errors (no response received) still raise.
- **`run_one`/`run_all` never raise.** Every probe outcome — success, exception, timeout — becomes a `HealthRow` so `test-all` can render a complete table without partial-failure semantics. Status carries `ok=False` plus a structured `detail` for the failed cases.
- **`ProbeRateLimiter` exists but is NOT auto-wired into the runner.** The brief says probes are rate-limited; the right place to enforce is at the CLI / doctor layer, where we know the user's intent (interactive run vs. CI loop). The runner stays composable.
- **`HealthCheckable` is a `Protocol` with a class-level `name: str`.** Avoids tightly coupling provider/notifier base classes to the health-check abstraction. Any duck-typed object with `.name` and `.health_check()` is checkable.

## Chunk 4 — playbook sandbox (AST-walking template evaluator)

- **Validation and evaluation are SEPARATE PASSES.** `validate(tree)` walks the AST and raises on any disallowed node without computing anything. `_eval(node, ctx)` then assumes the AST is clean. The injection-corpus tests exercise validation alone, so a regression that only blocked something at evaluation would still fail the gate.
- **Whitelist-only AST:** `Expression`, `Constant`, `Name`, `Attribute`, `Subscript`, `Compare`, `BoolOp`, `UnaryOp`, `BinOp`, `IfExp`, `List`, `Tuple`. Anything else — `Call`, `Lambda`, comprehensions, `Dict`, `Set`, `JoinedStr` (f-string), `NamedExpr` (walrus), `Starred`, `Slice`, `Yield*`, etc. — falls through to `DisallowedNode`.
- **Names whitelisted to `ioc` and `steps` only.** Everything else (`os`, `sys`, `__builtins__`, `len`, `open`) hits `DisallowedName` at validation. Adding new top-level names requires editing `_ALLOWED_NAMES` deliberately.
- **Underscore-prefixed attribute and subscript keys rejected** — covers all dunder pathways (`__class__`, `__bases__`, `__mro__`, `__subclasses__`, `__getattribute__`, `__dict__`, `__init__`, `__globals__`, etc.) plus single-underscore "private" names. Belt-and-suspenders: `_access_member` re-checks at eval time.
- **Subscript keys must be string-literal `Constant` nodes.** No integer subscripts, no slicing (`x[1:2]`), no tuple subscripts (`x[1, 2]`), no computed keys (`x[name]`). Cuts off DoS via huge slices and integer-indexing-into-strings paths.
- **`BinOp` allowlist is `Add`, `Sub`, `Mult`, `Div`, `Mod` — numerics only, bools rejected.** No `Pow` (DoS via giant ints), no `LShift`/`RShift` (also DoS), no `BitAnd`/`Or`/`Xor`, no `FloorDiv`. String concatenation deliberately disallowed — playbooks build strings via templates, not arithmetic.
- **`UnaryOp` allowlist is `Not`, `USub` only.** No `UAdd` (`+x`, useless), no `Invert` (`~x`, surprising semantics).
- **List and Tuple literals must contain only `Constant` elements.** Prevents `[ioc.value]` from materializing arbitrary objects into membership-test contexts. The classic `if x in [name1, name2]` pattern uses string constants — no need for richer element types.
- **`bool + int` rejected even though Python allows it.** `isinstance(x, bool)` is checked explicitly before `isinstance(x, (int, float))` because `True + 1 == 2` is surprising and almost always a bug.
- **Templates use a non-greedy `{{ … }}` regex, no escape sequence.** `\{\{\s*(.*?)\s*\}\}` with `re.DOTALL`. Multiple substitutions in one template parse independently. Phase 0 doesn't support literal `{{` or `}}` in template output — the template language is ours, and we don't yet need it.
- **`evaluate` short-circuits empty/whitespace input as `ParseError` rather than letting `ast.parse` raise a confusing `SyntaxError`.** Friendlier failure mode for the playbook-author edge case.
- **Internal-invariant checks at eval time even after validation passes.** `if not isinstance(node.slice, ast.Constant): raise SandboxError("internal: ...")` — defends against a future validator regression silently changing the AST shape downstream consumers expect.
- **Member access only on `Mapping`-typed objects.** Non-dict context values (strings, numbers, lists) refuse attribute access at the leaf, so `ioc.value.upper_case` raises rather than crawling into Python's string methods. Combined with the dunder rejection, there is no path from a legitimate context value back into the host class hierarchy.

## Chunk 5 — registry mechanics (intel providers + notifications)

- **Single shared `Registry[T]` lives in `core/registry.py`.** Both `intel/providers/` and `notifications/` instantiate it as `PROVIDERS` / `NOTIFIERS`. Avoids duplicating identical decorator + lookup code in two places. The brief framed them as "the same pattern" — sharing the implementation makes that explicit.
- **Decorator-based registration over `__init_subclass__`.** The brief allowed either; explicit `@PROVIDERS.register("name")` is easier to grep, easier to test, and easier to disable for a single subclass without touching the base class.
- **`base.py` is conventionally skipped by `discover_modules`.** Holds Protocol definitions and shared dataclasses, never concrete registrations. Skipping it by name avoids accidental import-side-effect surprises.
- **Discovery imports modules by name; module-level `@register` decorators run as the side effect.** No special "register me" hook on the Protocol itself — every concrete class explicitly opts in.
- **Protocols are `runtime_checkable`** so `isinstance(x, IntelProvider)` and `isinstance(x, Notifier)` work in tests and in any future runtime introspection. The check is structural (presence of attributes/methods), not nominal — concrete classes don't need to inherit from the Protocol.
- **`name: ClassVar[str]` and `instance: str` on the Protocol.** `name` is the registry key (one per concrete class). `instance` is the per-object label (`"default"`, `"mandiant"`, `"soc-alerts"`, ...). The combination is what audit entries and rate limits key off.
- **`from_config(cls, instance, cfg)` classmethod** is the only sanctioned constructor. Takes the config-file dict; secrets are fetched separately via the secrets manager. Keeps the construction protocol uniform across all providers and notifiers.
- **`IntelRecord.tags` defaults to empty tuple, not list.** Frozen dataclass demands an immutable default; tuple is the natural choice and makes the record hashable too.
- **`NotifyPayload.context: Mapping[str, Any]`.** Caller filters the playbook context by the notifier's redaction allowlist before constructing the payload. The notifier itself trusts that what it receives is already redacted.
- **`Notifier.send` returns `NotifyResult` and is contracted to never raise on transport failure.** Failure becomes `delivered=False, detail="..."`. The playbook engine treats notifier failure as a non-fatal step outcome by default.
- **`Registry.clear()` is documented "tests only".** Used by the `_clean_plugin_registries` autouse fixture in `conftest.py`. Production code should never call it.
- **Phase 0 ships zero concrete providers and zero concrete notifiers.** `discover()` returns `[]`, the registries are empty. Tests verify the mechanics by registering synthetic in-test classes; concretes land in Phases 1, 5, 6 per §8 of the brief.

## Chunk 6 — CLI + Textual shell + AI bridge skeleton

- **Hand-rolled minimal TOML writer in `core/config_io.py`.** Stdlib `tomllib` only reads. `tomli-w` would have been the ergonomic choice; for Phase 0 we hand-roll the small subset the wizard produces (scalars, nested tables, string lists). If a future schema needs arrays-of-tables / multi-line strings, switch to `tomli-w` rather than expanding the writer.
- **Atomic config writes via tempfile + `os.replace`** with the same restrictive-ACL apply path used by the audit log and secrets file. Permissions are enforced on both the tempfile and the renamed final file.
- **Doctor treats "not yet created" as OK, not FAIL.** A fresh install has no root dir, no config, no audit log, no secrets file. Doctor reports each as `OK / not yet created` so the first run after install is a clean green table even before `config` is invoked.
- **Doctor refuses to fail on keyring unavailability** — the encrypted-file fallback is a supported alternative, not a degraded mode. The check reports which backend is in play (e.g., `WinVaultKeyring`) so the user understands their secrets path. Permissive ACLs ARE a hard failure.
- **Doctor advisory-checks `claude` on PATH** but doesn't fail if absent. The AI bridge's clipboard fallback (Phase 4) means the tool is still usable without `claude`. The check reports the resolved path so users know whether Phase 4 will work out of the box.
- **`icacls` invocation extended to strip `Administrators` and `OWNER RIGHTS`.** Python's `mkdir` on Windows seeds new directories with EXPLICIT (not inherited) ACEs for those principals, which `/inheritance:r` does not remove. Without the explicit `/remove:g`, the verifier rightly rejects the dir as having unexpected principals. The fix was caught only because the doctor test expected a clean DACL on a `tmp_path`-created directory.
- **`(OI)(CI)F` inheritance flags on directory grants.** The first cut at the icacls fix used `:F` (this-object-only), which broke sqlite WAL writes because new files inside the directory didn't inherit the user's permissions. Files use `:F`; directories use `:(OI)(CI)F` so children inherit the owner+SYSTEM ACEs.
- **CLI uses Typer sub-apps** (`config`, `audit`) for namespacing. `secops-term tui` launches the Textual app; bare `secops-term` shows help (per Typer's `no_args_is_help=True`). Avoids the surprise of "I typed `secops-term` and now I'm in a TUI I can't escape from".
- **`secops-term config <subcommand>` plus a top-level interactive wizard.** Direct subcommands (`config chronicle`, `config vision-one`, `config intel <name>`) are scriptable; the bare `config` invocation prints the available subcommands. Phase 1+ can layer a richer interactive menu without breaking the scriptable surface.
- **Wizard stores secrets via `SecretsManager` (keyring or fernet fallback) and writes non-secrets via `config_io.save_config`.** The Chronicle service-account JSON, the V1 API token, and intel provider tokens all go to the keyring. Region pickers, allow_write toggles, and per-instance enabled flags go to `config.toml`. Matches §3.5.13.
- **`config test` and `config test-all` are placeholders in Phase 0** because no concrete providers exist yet to probe. The CLI surface is in place; Phase 1+ wires up real `health_check()` calls. Same for `config rotate`.
- **Textual app uses `SCREEN_BY_KEY` mapping** instead of `Binding` arguments referring to screen instances, so the bindings dispatch to a single `action_go(key)` method that pushes the right screen. Avoids constructing 8 screen instances at app startup.
- **`BINDINGS: ClassVar[list[BindingType]]`** annotation on the Textual `App` subclass — required by ruff `RUF012` (mutable-default class-attribute) and is the documented Textual pattern.
- **All 8 nav screens share a `_StubScreen` base class.** Phase 0 places nothing useful on any of them — concrete UIs land in Phases 1-5. The base class composes `Header + Static placeholder + Footer` so the navigation flow is exercisable end-to-end (and visible in screenshots).
- **AI bridge: Phase 0 ships only Transport A (`HeadlessClaudeBridge`).** Transport B (MCP server) and Transport C (clipboard) land in Phase 4. The Protocol exists so playbook engine code in Phase 5 can depend on it without import-cycle pain.
- **Subprocess hardening per §3.5.8 fully wired.** `asyncio.create_subprocess_exec` only, pinned absolute path, minimal env (only `PATH`, `HOME`/`USERPROFILE`, `APPDATA`, `LOCALAPPDATA`, `CLAUDE_CODE_GIT_BASH_PATH`), 60s default timeout (max 300s, kills process), 10 MiB stdout/stderr cap. Tests use a `_FakeProcess` that satisfies the same interface.
- **`build_subprocess_env` strips everything outside the allowlist** — including `AWS_*`, `MY_SECRET_TOKEN`, etc. Tested with synthetic env-var leaks to confirm they don't reach the child.
- **`wrap_untrusted` uses random per-call sentinel IDs** via `secrets.token_hex(8)` so a templated injection attack can't pre-construct a closing tag that matches our open tag. Tested by asserting two consecutive wraps of identical input produce different sentinels.
- **`--output-format json` schema is tolerated loosely** in Phase 0: we accept either `result` or `text` keys at the top level. The brief flagged that the exact schema needs verification against current Claude Code docs; this lenient parse keeps Phase 0 moving without committing to a specific shape that could change.

## Chunk 1.1 — IOC dataclasses + normalization + SQLite store

- **Normalization happens at upsert time, not at extraction time.** Providers emit `IntelRecord(type=..., value="raw")`; the store's `upsert` runs `normalize_value(type, value)` before the SQL `INSERT`. This way the `UNIQUE(type, value)` dedup key is stable regardless of which provider observed the IOC first or in what casing.
- **`refang()` is idempotent** — applying twice produces the same string. Tests assert this so the pipeline can re-refang without double-conversions producing surprising results.
- **`_canonical_url` lowercases scheme + host but preserves path/query/fragment verbatim.** Web paths are case-sensitive (and can leak case-significant tokens); collapsing them would merge unrelated URLs. Default ports (80/443/21) are dropped, non-default ports are kept.
- **Standalone domain extraction is deferred.** `iocextract.extract_domains` is too noisy for Phase 1 (catches `e.g.`, "Trend Micro Research", etc.). Domains enter the store via URL-host extraction or explicit provider-side typing (e.g. abuse.ch ThreatFox returns explicit `domain` records). A stricter dedicated domain-from-text extractor can be added later if a provider needs it.
- **CVE detection is regex-based, not via `iocextract`.** `iocextract` doesn't ship a CVE extractor; we use `^CVE-\d{4}-\d{4,}$` (case-insensitive on input, uppercased on output).
- **Hash type is determined by length, not heuristics.** 32 chars → md5, 40 → sha1, 64 → sha256. `iocextract.extract_md5_hashes` / `extract_sha1_hashes` / `extract_sha256_hashes` keep these straight at the producer side too.
- **`IOCStore.upsert` returns `(ioc_id, is_new_observation)`.** Callers (the playbook engine in Phase 5) need to know when to enqueue retro-hunt jobs — only on first observation, not on every re-fetch from a different provider.
- **`COALESCE(?, confidence)` on update** preserves a previous higher-confidence value when a re-observation has `confidence=None`. A real value will always overwrite (because COALESCE picks the first non-null arg).
- **Method named `find` rather than `list`.** Under `from __future__ import annotations`, mypy resolves return-type strings against the class scope, where a `list` method shadows the `list` builtin and breaks `-> list[IOC]` annotations. `find` sidesteps the conflict cleanly.
- **`search` escapes SQL `LIKE` wildcards.** `_` and `%` in user queries are escaped (`\_`, `\%`) so they match literally. Without this, a user searching for `a_b` would match `axb` because `_` is the LIKE single-char wildcard.
- **`find` and `count` enforce `limit ∈ [1, 10000]`** to prevent runaway queries from blowing memory. The bound is generous for legitimate use; a UI that needs more should paginate via `offset`.
- **Migration `001_intel.sql` ships three tables + four indexes** (`iocs`, `ioc_sources`, `retro_hunt_jobs` per §6.1). `ON DELETE CASCADE` on `ioc_sources` and `retro_hunt_jobs` so a future "delete this IOC" command cleans up child rows automatically — `PRAGMA foreign_keys=ON` (set at connection open in Phase 0) is required for the cascade to fire.
- **`# noqa: S608` on the dynamic SELECT in `find`** — composition uses only constant column-name fragments; user values are bound via `?` placeholders. ruff's pattern-based check can't see that, so the noqa carries an explanatory comment.
- **`migrated_db` pytest fixture in `conftest.py`** opens a fresh `Database` and runs `apply_migrations(discover_migrations())` before yielding. Tests get a guaranteed-clean DB with all checked-in migrations applied; teardown closes the connection.

## Chunk 1.2 — scraper / IOC-extraction pipeline

- **`requests` added as a direct dep.** `iocextract` 1.16.1 imports `requests` at module load but doesn't reliably declare it as a dependency. Pinning `requests>=2.31` ourselves keeps installs reproducible and avoids the silent "ModuleNotFoundError: No module named 'requests'" at first use.
- **Refang the input text once, then extract.** Running every iocextract per-type call without the `refang=True` flag and pre-canonicalizing produces a single normalized result regardless of whether the source text used `hxxp://` or `http://`. Side benefit: deduplication within the call is automatic since both forms collapse to the same canonical value.
- **Replaced `iocextract.extract_emails` with our own `\b…@…\.tld\b` regex.** iocextract's email extractor is greedy with leading whitespace ("Contact attacker@x.com" came back as a single match including the leading word). Our word-boundaried regex matches what `_canonical_email` validates downstream — the round trip is now consistent.
- **Standalone-domain extraction stays out of the scraper.** `iocextract.extract_domains` produces too many false positives for free-form intel writeups (catches "e.g.", "Trend Micro Research" as domains). Domains enter the corpus via URL hosts and explicit provider-side typing. Documented in `extract_iocs_from_text`'s docstring so future maintainers don't add it back without thought.
- **CVE detection is regex-based** (`\bCVE-\d{4}-\d{4,}\b`, case-insensitive). iocextract has no CVE extractor; our regex is anchored on word boundaries to avoid catching `…CVE-2024-1234some-other-text…` substrings as part of a longer ID.
- **Each iocextract call is wrapped in `try/except: continue`.** A bug in one extractor (e.g., a regex that fails on an edge-case input) must not break the others. ruff `S112` (try/except/continue) is acknowledged with a `# noqa: S112` and a justifying comment — silent swallow is intentional here because every extractor is independent and the overall extraction pipeline must be resilient.
- **`extract_text_from_html` strips `script`/`style`/`noscript`/`nav`/`footer`/`header`/`aside` before pulling text.** This avoids navigation IOCs (URLs in nav menus), inline-script "IOCs" that aren't actually relevant content, and CSS/style noise. Handles malformed HTML gracefully — `selectolax` is permissive and we fall back to the root element if `body` is missing.
- **`RobotsChecker` defaults to "allow on uncertain"** — HTTP non-200 (no robots.txt published, server error) and any network exception are treated as no-rules-published. This matches industry convention for non-crawler clients: a robots.txt that can't be fetched isn't disallowing anything specific. The brief specifies "respect robots.txt by default", which we do; it doesn't specify the unreachable-robots case.
- **`robots.txt` cache is per-origin with 1h TTL.** Origin = `scheme://host[:port]`. A small race exists where two concurrent `allowed()` calls for the same origin may both miss the cache and both fetch — that's harmless since both writes produce identical state.
- **Per-source robots opt-out via `respect_robots=False`** is allowed but logged. The scraper emits a `scraper.robots_bypassed` audit entry whenever a caller bypasses the check, satisfying the brief's "explicit user opt-in **and** logged" requirement. Same pattern for `scraper.robots_disallowed` (when robots blocks a fetch) and `scraper.fetched` (every successful scrape, with IOC count).
- **Snippet (±200 chars) is captured from the refanged text, not the original.** Phase 1 trade-off: we lose the exact original wording in favor of consistent context that lines up with the canonical IOC value. If a downstream tool needs the original, the source URL is preserved in `source_ref` and the original can be re-fetched.
- **Scraper uses the original URL, not the pinned IP, for the actual fetch.** Brief §3.5.4 calls for transport-level pinning (custom httpx transport that connects to IP with hostname for SNI/TLS) — that's deferred to Phase 2 alongside the first real provider client. For Phase 1 we still validate via `url_guard` (rejects private/loopback/link-local) but the connection itself uses the hostname. The TOCTOU window between validation and connection is acceptable for v1; documented for Phase 2 to close.

## Chunk 1.3 — three concrete intel providers (`abuse_ch`, `otx`, `rss`)

- **`Registry.snapshot()`/`Registry.restore()` added** as test-only helpers. Phase 0's autouse `clear()`-between-tests fixture wiped concrete provider/notifier registrations because Python's import cache means the `@register` decorator only runs once per session — re-importing won't re-decorate. The new fixture pattern: a session-scoped fixture runs `discover()` once, then a per-test fixture takes a snapshot at entry and restores at exit, preserving concretes while wiping any synthetic in-test registrations.
- **Providers use `secrets.get_manager()` (singleton) for tokens, not constructor DI.** This keeps the `from_config(cls, instance, cfg)` Protocol signature unchanged from Phase 0. Tests inject a fake keyring module via `SecretsConfig.keyring_module` instead. Slightly less explicit than DI but keeps the brief's contract intact.
- **Each provider creates its own `HardenedClient` per `pull()` / `health_check()`.** No connection pooling in Phase 1 — slightly less efficient but isolates failures and simplifies lifecycle. Tests hit this via `respx`, which patches at httpx's transport layer regardless of how many client instances exist.
- **abuse_ch: 4 sub-feeds, all under one provider.** A single `auth.abuse.ch` API token authenticates URLhaus, MalwareBazaar, ThreatFox, and Feodo Tracker. Per-instance config has a `sub_feeds` list with the four valid keys; missing list → all four enabled. Unknown sub-feeds at construction time → `IntelProviderError`.
- **abuse_ch sub-feed isolation: one feed's HTTP failure doesn't block others.** Each sub-feed puller catches its own exceptions and returns `[]` on failure. The pull-level result is the union of whatever each feed produced — partial success is preferred over all-or-nothing.
- **abuse_ch ThreatFox `ip:port` → `ipv4`.** ThreatFox sometimes returns C2 IOCs as `1.2.3.4:443`; we strip the port before storing because the `iocs` schema is type-keyed on `ipv4` (not `ipv4:port`). The port is lost, but it would only have been recoverable as `context` anyway.
- **abuse_ch health probe = ThreatFox `get_iocs` with `days=1`.** Auth-validating, quota-cheap, and exercises the same auth path as the real pulls. Returns 401 on bad token, `query_status="ok"|"no_result"` on success.
- **otx hostname → domain.** AlienVault distinguishes `domain` and `hostname`; our schema only has `domain`. `hostname` values include subdomain prefixes (e.g. `mail.evil.example.com`) which `_canonical_domain` accepts as-is.
- **otx unknown indicator types are silently dropped.** OTX adds new types over time (BTCAddress, Mutex, etc.). Anything not in `_TYPE_MAP` gets skipped without errors so a new OTX type can't break the whole pull.
- **otx health probe = `/users/me`.** Per brief §3.5.14 — quota-free for the auth user. 200 → ok with username. 401/403 → auth rejected. Anything else → generic failure.
- **rss is multi-instance, summary-extraction by default.** Each named instance is one feed URL. Phase 1 default is summary-only (extract IOCs from each entry's title + summary text). `scrape_articles=true` opts into fetching each entry's `link` and running the full scraper over the article body, with `respect_robots` controlling the robots.txt check.
- **rss feed text is HTML-stripped if it looks like markup.** Feed `summary` content is often HTML — we run it through `extract_text_from_html` when `<` and `>` are both present, otherwise treat it as plain text. Avoids extracting URLs/IOCs from `href`/`src` attribute scaffolding when the summary is already plain prose.
- **rss `since` filter uses the entry's `published_parsed`/`updated_parsed`.** feedparser returns these as 9-tuples; we take `[0:6]` and explicit-unpack into `datetime(year, month, day, hour, minute, second, tzinfo=UTC)` so mypy doesn't get confused by `datetime(*tuple, tzinfo=UTC)`.
- **rss provider rejects private feed URLs.** `url_guard.validate_url(self._feed_url)` runs at the top of `pull()` and `health_check()`. Private/loopback/CGNAT URLs surface as `URLGuardError` (in `pull`) or as `ok=False` with a "rejected" detail (in `health_check`).
- **One bad article doesn't kill the rss pull.** When `scrape_articles=true`, individual article fetches that fail (URL-guard rejection, robots disallow, 4xx, etc.) return `[]` for that article. The summary IOCs still flow through, and other articles' bodies still get scraped.
- **`S107` added to per-file-ignores for tests.** The fake-keyring helper functions in provider tests use a default `token="test-..."` arg as a documentation hint; ruff flags it as a hardcoded password. False positive — added to the same ignore list as `S101`/`S105`/`S106`.
- **Existing `test_phase_0_ships_no_concrete_providers` test renamed and updated.** Phase 1 ships three concretes; the test now asserts `{"abuse_ch", "otx", "rss"} ⊆ discovered names`. The notifier equivalent (`test_phase_0_ships_no_concrete_notifiers`) stays valid because notifier concretes ship in Phase 5.

## Chunk 1.4 — orchestrator + `intel pull` CLI + `config test` wiring + Threat Intel screen

- **`intel/orchestrator.py` is the single integration point** between config, registry, providers, and store. Both the `intel pull` CLI and the `config test-all` CLI go through it (one walks-pulls-upserts, the other walks-builds-targets). Keeps the CLI thin and lets the orchestrator be tested in isolation against synthetic providers.
- **`PullResult.error` is set on per-pair failure; `pull_all` does NOT raise.** A bad provider returns its own `PullResult(error=...)` and the others still run. The CLI exits non-zero if any result errored, but only after writing every successful pull's records to the store.

- **`pull_all_sync` for the CLI** wraps the async function in `asyncio.run`. Typer commands run synchronously, and centralizing the loop construction in one place avoids each CLI subcommand having to spell out `asyncio.run(...)`.
- **`HealthCheckable.name: ClassVar[str]`** (was bare `str`). Concrete providers/notifiers set `name` at class level (the brief's pattern); under `mypy --strict` a class-level `name = "..."` matches a `ClassVar[str]` Protocol attribute but doesn't match a bare instance attribute. Updating the Protocol fixes the type error without changing any concrete class.
- **`SCREEN_BY_KEY` is typed `dict[str, type[Screen[None]]]`** (was `type[_StubScreen]`). Phase 1 ships a real `IntelScreen` that subclasses `Screen` directly — widening the dict type accommodates that and any future real screens without forcing them to inherit from the stub base.
- **`IntelScreen` lazily opens the IOC store** the first time `_get_store()` is called rather than in `__init__`. This keeps construction side-effect-free (helpful for screen-instantiation in tests) and means a screen that's never displayed never touches the disk.
- **`IntelScreen` calls `store.search()` for non-empty queries, `store.find()` otherwise.** The branch is intentional: `find` is type-filterable but doesn't substring-match; `search` substring-matches but ignores type filters. Phase 1 keeps these two paths independent — Phase 2+ may merge them once the screen needs combined search-and-filter.
- **`config test <provider>` is a positional-argument command, not optional.** Phase 0's placeholder accepted no args ("would print Phase 0 message"); Phase 1 requires a provider name because health-checking implicitly-everything is what `config test-all` exists for. The placeholder test was updated to assert exit-code-not-zero on missing argument (Typer's standard behaviour for missing positionals).
- **`intel list` opens its own store and closes it in a `finally`.** CLI commands are short-lived; not closing the connection would leak a file handle until process exit (fine in practice, but the explicit close is cheap and matches how the orchestrator manages store lifetime).
- **The orchestrator's two `try/except` swallow blocks are tagged `# noqa: S112`.** Per-record upsert failure (in `_pull_one`) and per-instance `from_config` failure (in `build_health_targets`) are both intentionally silent: pull surfaces individual failures via the `PullResult.error` field, and health-target construction is a non-essential pre-step for `test-all` that defers actual error reporting to doctor / pull. Comments explain.
- **Synthetic-provider test pattern** (`@PROVIDERS.register("test-name") class _Whatever`). Tests register a synthetic provider inside the test body; the autouse `_isolate_plugin_registries` fixture (added in Chunk 1.3) snapshots at entry and restores at exit, so the synthetic registration is wiped after the test without affecting the three concrete providers (`abuse_ch`, `otx`, `rss`) that the session-scoped discover put in place.

## Chunk 2.1 — Chronicle auth + UDM Search client

- **`google-auth>=2.0` added as a direct dep.** Chronicle's service-account → access-token flow uses Google's stock library; we wrap their sync transport in an `asyncio` executor so the caller's loop stays responsive during refresh. Fully optional at import time — the import is lazy inside `GoogleServiceAccountAuth._build_credentials`, so users who never wire up Chronicle don't pay for it (and the module imports without the package installed, raising a friendly `ChronicleAuthError` only on first use).
- **Two `ChronicleAuth` implementations, one Protocol.** `GoogleServiceAccountAuth` is production; `StaticTokenAuth` returns a fixed bearer for tests. Both satisfy the `ChronicleAuth` Protocol so `ChronicleClient` doesn't care which one it gets — same auth contract whether you're hitting the real API or respx-mocked tests.
- **Service-account JSON accepted as `dict` OR `str`.** When the secrets manager pulls the value out of keyring it comes back as a string; `GoogleServiceAccountAuth.__init__` accepts either, calling `json.loads` on strings. Saves the caller from doing the same parse at every site.
- **`credentials_factory=` injection point on `GoogleServiceAccountAuth`.** Tests bypass real `google-auth` by passing a factory that returns a fake credentials object — the only-touchpoint pattern means we don't have to mock `google.oauth2.service_account.Credentials.from_service_account_info` globally and we never need to ship a real service-account JSON for tests.
- **Refresh runs in `loop.run_in_executor(None, ...)`.** `google-auth-requests` is a sync transport that blocks during the network call. Running it in the default executor keeps a single CPU-bound thread off the event loop. The token-cache check (`creds.valid` + `creds.token`) means we only pay this cost when the cached token is invalid.
- **Region → base URL map covers nine documented Chronicle SecOps regions.** `ChronicleConfig.resolved_base_url()` uses the map by default; `base_url=` overrides for private deployments / future regions. Unknown region without an override raises `ChronicleError` rather than silently producing a broken URL.
- **Trailing slash stripped from `base_url` overrides** so the path concatenation (`{base_url}/v1alpha/...`) doesn't produce double slashes.
- **`ChronicleClient.udm_search_url()` is `{base_url}/v1alpha/{customer_id}:udmSearch`.** Phase 2.1 keeps it deliberately simple — Chronicle API paths have shifted across versions and we'd rather have a one-line override (set `base_url=` to point at a private deployment) than try to model every variation upfront.
- **UDM Search request body matches Chronicle's documented shape:** `{"query": "...", "time_range": {"start_time": ISO, "end_time": ISO}, "limit": N}`. `start_time` is computed from `lookback_hours` so callers don't have to do the arithmetic themselves; pass `end_time=` to override.
- **Validation upfront on `udm_search()`:** empty query → `ChronicleError`; `limit < 1` → `ChronicleError`; `lookback_hours < 1` → `ChronicleError`. These checks land before any network call so misuse fails fast and quota-cheap.
- **`UdmSearchResult` keeps `events` as `list[dict[str, Any]]`** for Phase 2.1. Phase 2.3 will narrow once the Retro Hunts screen needs specific UDM fields. `raw` stores the whole payload so consumers can pull anything we haven't surfaced yet.
- **`ChronicleAPIError` carries the status code separately** so the health probe can distinguish 401/403 (auth) from 5xx (transient) without parsing the message string. Health probe converts each into a different `HealthStatus.detail`.
- **Health probe = real UDM search with `principal.ip = "0.0.0.0"`, lookback 1h, limit 1.** Auth-validating, exercises the same code path as production searches, returns 0 events from real telemetry. Phase 2.3 may switch to a dedicated Chronicle metadata endpoint if the user prefers, but for now the search-with-empty-filter approach also catches misconfigured customer IDs and bad URL templates.
- **`ChronicleClient` opens its own `HardenedClient` per call** with a 50 MB response cap (UDM searches can return large payloads). Same pattern as the intel providers in Phase 1 — no shared HTTP state across requests, simpler lifecycle, slightly more TLS overhead but easy to optimize later.

## Chunk 2.2 — UDM query builder + retro-hunt worker

- **Query builder is a pure function on `(ioc_type, ioc_value)`.** No store access, no I/O. Tests don't need a DB, the worker imports it freely, and any caller (the playbook engine in Phase 5, ad-hoc CLI tools later) can build queries without instantiating a Chronicle client.
- **UDM expressions are bare filters, no `rule {}` wrapper.** Per the user's documented preference for the Chronicle UDM Search syntax. Each IOC type maps to one expression with `OR`-combined fields per brief §6.2.
- **`_escape()` runs unconditionally on every value.** `normalize_value` already rejects values containing quotes for every supported type, but the escape is belt-and-suspenders defence against future code paths that bypass normalization (or against bugs that let a tainted value through). Backslash and double-quote get the standard `\\` / `\"` treatment.
- **Email and CVE shipped despite not being explicit in the brief table.** Brief §6.2 lists ipv4/ipv6, domain, url, hashes, and CVE; we add email mapping (`network.email.from = X OR network.email.to = X`) because it's the obvious analogue and tests already exercise the email IOC type end-to-end. CVE maps as the brief specifies; the "product fallback" hinted at in §6.2 is a future enhancement.
- **`UnsupportedIOCType` is a separate exception, not a `ChronicleError` subclass.** It signals "this IOC isn't routable to Chronicle", which is a per-job decision (skipped/failed) distinct from "Chronicle is broken". The worker classifies these as `skipped` rather than `failed` in `RunResult`.
- **`retro_hunt_jobs` job state machine is simple: queued → running → done | error.** No retry/resume yet — a failed job stays `error` until something explicitly re-enqueues. Phase 5 (playbooks) may add a `retry` action; for Phase 2.2 the user can enqueue manually if they want to retry.
- **`next_pending_job` is a single-row claim wrapped in `BEGIN IMMEDIATE`.** Atomic state transition from `queued` → `running` so two concurrent workers can't pick up the same job. Single-process tool means it's mostly belt-and-suspenders, but doesn't cost anything and avoids surprising behaviour if the playbook engine later spawns parallel workers.
- **`fail_job` truncates error messages to 2000 chars.** Keeps a single Chronicle stack trace from blowing up the SQLite row size and makes recent_jobs queries fast even when something went catastrophically wrong on the API side.
- **Worker catches at three levels: `ChronicleAPIError` → "HTTP NNN" detail, `ChronicleError` → generic message, `Exception` → `TypeName: message`.** Each goes through `fail_job` with the appropriate detail; the worker never crashes from a per-job failure. `RunResult.{drained, succeeded, failed, skipped}` lets the caller present a single summary.
- **`worker.run_once(max_jobs=N)` is the unit of work**, not `run_forever()`. Background polling lives at the CLI / scheduler layer (Phase 2.3 wires it up). Keeps the worker testable end-to-end without involving any time/sleep machinery.
- **`hits` falls back to `len(result.events)` when `total_events` is missing.** Chronicle's UDM API is documented to return `total_events`, but real responses sometimes omit it (especially on edge accounts) — the fallback keeps the job-summary count meaningful even then.
- **Worker forwards `lookback_hours` and `limit_per_query` to the UDM call.** Defaults: 30 days lookback (matches brief §6.2) and 1000 events per query. Both per-instance configurable so a user with quota concerns can tighten the lookback or page through smaller batches.
- **Test file naming: `tests/intel/test_store_retro_hunt.py` (CRUD + claim semantics) and `tests/chronicle/test_retro_hunt.py` (query builder + worker).** Two files instead of one because the store-side tests don't need Chronicle (no respx, no async client) — fast pure-SQLite tests run in <100ms each, separate from the worker tests that mock Chronicle.

- **Tolerant response parsing.** `computers` (agents) / `alerts` keys may be absent on edge responses → empty list. `totalCount` / `total` (DS uses both spellings across product lines) → optional. Full payload always preserved in `raw=` for the alert-ingest layer.
- **`DS_KEY_FIELD = "api_key"` does NOT need `# noqa: S105`.** ruff doesn't flag `"api_key"` (only `"password"`-ish substrings); we noticed because we'd preemptively added the noqa and ruff's `RUF100` rejected it as unused. Compare with `V1_TOKEN_FIELD = "api_token"` which DOES contain "token" and trips S105.

## Chunk 3.3 — unified Alerts pipeline + screen + CLI (Phase 3 finale)

- **`Alert` is the unified shape — frozen dataclass, source-tagged, `raw=` preserved.** `id`, `source` (Literal["chronicle","vision_one","deep_security"]), `severity` (5-level Literal), `title`, `detected_at` (UTC datetime), `entities` (tuple), `raw` (original payload), `dedupe_key` (`{source}:{id}`). Frozen so groups/dedup can hash them; `raw=` preserved so downstream playbooks (Phase 5) can reach back into source-specific fields without changing this layer.
- **5-level severity scale: `info / low / medium / high / critical`.** Maps each vendor's idiosyncratic scale (Chronicle "Informational" → info, V1 "MEDIUM" → medium, DS numeric 80 → high) to a single ordering. Unknown severity falls back to `medium` rather than dropping the alert — better to surface than silently lose.
- **DS numeric severity bucketing: ≥90 critical, ≥70 high, ≥40 medium, ≥20 low, else info.** Aligns with the typical DS 1-100 score where 90+ is "high alert / threat detected" and 1-20 is informational. Strings still take priority via `_DS_SEVERITY_STRING_MAP` so a DSaaS payload sending `"severity": "high"` Just Works.
- **`_coerce_str` accepts `int|float`.** DS sometimes returns `"ID": 42` as an integer; we coerce to `"42"` to keep the dedupe_key shape stable. `bool` is explicitly rejected (Python booleans are ints — `True` is not a meaningful id). Without this, DS alerts fell back to a sha256 hash id even when a real numeric id was present.
- **`Entity` is `(type, value)` only.** No `confidence`, no `attributes`. Source-specific extra fields stay in `Alert.raw`. Canonical entity ordering for "primary entity" picks: `host > user > ip > domain > url > file > process > email`. The first match across the alert's entity tuple wins; this drives the grouping signature.
- **Per-source entity extractors target the documented payload shapes.** Chronicle: `principal/target/src/dst.{ip,hostname,user.userid}` plus a generic `entities[]` walker. V1: `impactScope.entities[].{entityType,entityValue}` plus top-level `endpointHostName` / `ipAddress`. DS: `computerName` / `hostname`. Unknown entity types are silently skipped — better to lose a weird type than fail the whole alert.
- **`title_signature(title)` for grouping: lowercase + strip digits + collapse whitespace.** "Login failure user-123 at 10:00" and "Login failure user-456 at 10:30" reduce to the same signature, so they cluster instead of appearing as N near-duplicates. Digits are replaced with `#` rather than removed, so "host123" doesn't merge with "host" — the structure of the title still matters.
- **Dedup uses `dedupe_key` only — first-seen wins.** Same `{source}:{id}` collapses to a single alert, regardless of severity / timestamp differences in successive ingests. Distinct sources with the same id stay separate (different prefix). `dedupe_alerts` preserves first-seen order so the caller's input order is meaningful.
- **`group_alerts` clusters by `(title_signature, primary_entity)` within a 1-hour window.** Sort newest-first; bucket; the newest alert in each bucket becomes the `representative`. Configurable via `window=` kwarg. Empty input returns `[]`. Different entities → different groups even with the same title; same entity outside the window → different groups even with the same title.
- **`_fallback_id` is a sha256 of the sorted-items repr of the payload.** Stable across runs given the same payload, distinct across different payloads. Used when a source omits an id field — Chronicle alerts without `id`/`name`, V1 without `id`. Prefix is `{source}-` so it's still grep-able.
- **`_parse_dt` tolerates `Z` suffix and missing timezone.** Strips trailing `Z`, replaces with `+00:00`, falls back to `now(UTC)` on parse failure. Naive datetimes get UTC tagged. Aligns with `audit.iso_now_utc()`'s convention from Chunk 2 — every timestamp in the system is timezone-aware UTC.
- **`ingest_all(cfg_data, since=None, limit=None)` is the single entry point.** Reads which sources are configured from cfg_data; calls each enabled source concurrently-but-not-actually (sequential `await` for now — concurrency adds complexity that pays off only with 5+ sources). Returns `IngestResult(per_source, alerts, groups)` so callers can show per-source health alongside the unified list.
- **Per-source failures are isolated, not fatal.** Each `_ingest_<source>` is wrapped in `try/except Exception` that returns a `SourceResult(source, alerts=[], error=str(exc))`. One source 500'ing doesn't drop the alerts from the other two. The CLI surfaces this via the `per_source` list; the screen shows it in the status line as `chronicle=err · vision_one=12 · deep_security=3`.
- **`since=` is forwarded to all three sources** with each source's parameter name: Chronicle `startTime=`, V1 `startDateTime=` (mapped to `start_time` kwarg), DS `since=`. ISO-8601 UTC; the test asserts the actual outbound URL contains the right query param for each. Phase 5 playbooks will use this for incremental sync.
- **Chronicle's `list_alerts` uses `{base_url}/v1alpha/{customer_id}:listAlerts`.** This is a placeholder — Chronicle's real Detection API path shifts across versions and the brief doesn't pin one. Documented inline; switching to the correct path is a one-line change when the customer's Chronicle tier is known. Returns `ChronicleAlertsResult(alerts=list[dict], next_page_token=str|None)` mirroring the V1/DS pattern.
- **Alert dataclass is `frozen=True`** so it can be hashed for set membership and dict keys. Required for the dedup pass (set of seen `dedupe_key`s) and the grouping pass (dict keyed by signature tuple). `raw=` is `dict[str, Any]` — tolerated as non-hashable because `Alert.__hash__` would have to ignore it; instead, we don't hash Alerts directly, we hash their `dedupe_key`. Saves an `__eq__` / `__hash__` override.
- **CLI: `secops-term alerts list`** with `--source chronicle|vision_one|deep_security`, `--severity info|...|critical`, `--since YYYY-MM-DDTHH:MM:SS`, `--limit N`, `--grouped`. Default is raw newest-first. `--grouped` switches to the group view. Per-source error lines render in red so a partial outage is visible at a glance.
- **`AlertsScreen` (Textual)** is the alerts UI binding to `r` (refresh) and `g` (toggle group). Same DataTable + Static-detail pattern as Threat Intel and Retro Hunts (Phase 1.4 / 2.3). Refresh is async — ingestion calls go out concurrently, status line updates inline so the user sees "Ingesting..." → result. Detail panel shows entities, dedupe_key, source-specific raw fields on row select. `escape` returns to nav.
- **`DataTable[str]` type parameter** added to satisfy mypy strict's `type-arg` rule. Cells are str-only here; if we ever stuff Rich renderables in (color-coded severity), the parameter widens. Cheap fix vs. a global mypy override.
- **`doctor` extended to V1 and DS — config-only checks, no live probes.** Same pattern as Chronicle from Chunk 2.3: verify the config block exists (or doesn't), verify the keyring entry is present, verify `build_*` returns successfully. Live probes still live behind `config test <provider>` / `config test-all`. Keeps `doctor` fast and offline-safe.
- **`build_health_targets()` extended to V1 and DS** — the orchestrator's health-target builder now yields up to three Chronicle/V1/DS targets in addition to the intel providers. Lazy imports inside the function avoid a circular path between `intel/orchestrator.py` and the trendmicro factories.
- **`config test vision_one` / `config test deep_security`** are special-cased in the CLI alongside Chronicle, not pushed through the intel-providers walker. Each non-intel provider has its own config block, its own factory, and its own health-probe path; treating them as special cases keeps the call site honest about that.
- **`tmv1_query` builder for V1 retro-hunt is in `secops_term/trendmicro/v1_retro_hunt.py`** (constant `VISION_ONE_PLATFORM = "vision_one"`). Only the query builder lands in 3.3 — wiring V1 as a hunt-worker platform is deferred to Phase 6 polish. The brief allows enqueueing V1 hunts through the existing queue (`hunt enqueue --platform vision_one`); the worker side just doesn't exist yet.
- **`tests/alerts/` covers normalize, dedup, ingest end-to-end** — 37 tests. respx mocks the three vendor APIs; `fake_secrets_all_three` fixture seeds keyring entries for all three providers; `patch_chronicle_creds` injects fake google-auth credentials so Chronicle calls don't hit the real OAuth dance. One-source-failure-doesn't-block-others is a dedicated test, ditto since-forwarding to all three sources.
- **AlertsScreen stub-removal in `ui/screens/__init__.py` is via import, same pattern as IntelScreen / RetroHuntsScreen.** Three real screens now shadow their Phase 0 stubs from the same `__init__.py`. The `_StubScreen` base remains for the four still-stub screens (Dashboard, Playbooks, Query, Config, Audit).

## Chunk 4.1 — AI bridge audit, clipboard transport, selector

- **`AuditingBridge` is a wrapper, not a mixin.** Decorates any `AIBridge` (the Protocol from Phase 0) and emits a `kind="ai_call"` audit entry per `complete()` call. Keeps each concrete transport (`HeadlessClaudeBridge`, `ClipboardBridge`, future MCP) focused on its mechanics; audit policy lives at one place applied at the outer layer where the user-facing call lands. Mirrors the `HardenedClient` ↔ `AuditLogger` relationship from Phase 0.
- **Audit entry shape locked: `{kind, transport, prompt_hash, response_hash, untrusted_count, latency_ms, ok}`.** No prompt or response text by default. `--debug-ai` (constructor flag `debug_ai=True`) adds `prompt` and `response` keys. Failure entries omit `response_hash` and add `error="<exception class>: <message tail>"` truncated to 200 chars.
- **`prompt_hash` is computed from a canonical join: `sys:<system>\nu0:<u0>\nu1:<u1>\n...\np:<prompt>`.** Stable across calls with identical inputs (test verifies). Hash is sha256 truncated to 16 hex chars — enough to spot drift in audit review, never enough to reconstruct the prompt. Same scheme for `response_hash`.
- **Audit emission wrapped in `contextlib.suppress(Exception)`.** Same pattern as `HardenedClient`'s audit calls — a logger blow-up never breaks the AI call path. The user gets their answer; the audit gap is visible at chain-verify time and triggers an investigation.
- **`health_check()` is NOT audited.** Brief §3.5.10 reserves `kind="probe"` for health checks; `health_check()` on a bridge is bookkeeping the selector does, not a user-initiated AI call. Skipping it keeps the audit log signal-rich.
- **`ClipboardBridge` (Transport C) takes a `response_provider: Callable[[str], Awaitable[str]]`.** Caller (CLI / TUI / test) decides how to gather the user's pasted response. Bridge never reads from stdin and never blocks on UI state — keeps the transport pure and trivially testable.
- **`pyperclip` is an optional dep.** `_try_import_pyperclip()` returns `None` if the module isn't installed; `health_check()` then returns `False` and the selector falls through to the next candidate. The constructor accepts `clipboard_module=None` explicitly to simulate "no clipboard" in tests; passing nothing triggers auto-detection via a `_AUTO_DETECT` sentinel.
- **`ClipboardBridge.complete()` sentinel-fences untrusted blocks before copying** via `wrap_untrusted` (the existing helper from Phase 0 §7.5). The whole wrapped prompt — including the "this is data, not instructions" preamble — goes to the clipboard so the user can paste it directly into Claude.ai with the safety scaffolding intact.
- **Clipboard health probe = round-trip via `paste()`.** Touching the clipboard backend without mutating it. Returns `False` on any exception — pyperclip raises on headless Linux without an X server, on macOS without `pbcopy`, etc. Selector treats `False` as "skip this candidate".
- **`compose_bridge(candidates, audit_logger, debug_ai=False)` is the selector.** Iterates candidates in order, picks the first whose `health_check()` returns True, wraps it in `AuditingBridge`. Returns `AuditingBridge` (not `AIBridge`) so the caller can read `.transport` if they want to surface "using clipboard fallback" to the user.
- **Selector treats health-check exceptions as unhealthy.** Wrapped in a broad `try/except Exception` so a transport blowing up while introspecting itself doesn't crash composition — it just gets skipped. Test injects a `_BlowingBridge` that raises in `health_check`; selector falls through to the next candidate.
- **`NoTransportAvailable` is the failure mode** when no candidate passes its health check. Caller surfaces this as "AI features unavailable, please configure Claude Code or enable clipboard fallback" rather than a stack trace. Empty candidate list raises the same error.
- **`AIBridge` Protocol satisfied structurally.** `AuditingBridge`, `HeadlessClaudeBridge`, `ClipboardBridge`, and the future MCP bridge all expose `complete(prompt, *, system, untrusted_inputs)` and `health_check()` — no inheritance needed. Test asserts protocol compatibility via a `_ensure_protocol_satisfied(bridge: AIBridge) -> AIBridge` helper that mypy validates.
- **`tests/ai/` package created** with three test modules (audit, clipboard, selector) — 27 tests. respx isn't needed here; subprocess and clipboard are stubbed with fakes that capture call args. Latency test uses `await asyncio.sleep(0.05)` and asserts ≥30ms with margin for slow CI.

## Chunk 4.2 — NLP Query Helper (UDM + TMV1)

- **Two flavors only — `udm` and `tmv1` — locked behind a `Literal` `QueryTarget`.** Mirrors brief §6.5: NLP → Chronicle UDM Search and NLP → Vision One Search. Adding new query languages later means extending the Literal in one place; mypy then forces every consumer to handle the new variant.
- **Prompt template = cheat sheet + few-shot examples + question + output contract.** Cheat sheet is platform-specific (UDM field paths vs. TMV1 camelCase fields); few-shots are 3-example placeholders today, must be replaced with 5-10 real examples from the user's UDM Search reference doc (the 14-investigation-categories doc) before this is shipped to operators. Each example is `(intent, query)` — Claude sees both so it learns the mapping.
- **`UDM_FEW_SHOTS` and `TMV1_FEW_SHOTS` are TODO placeholders.** Marked `TODO(user):` in module-level comments so a grep for `TODO(user)` surfaces all the spots awaiting input. The placeholder examples illustrate the expected shape — DNS query, login event, process launch — so the prompt is never empty even pre-handoff.
- **System prompt locks output contract: "Return ONLY the query string."** Claude tends to wrap responses in Markdown fences and add explanatory prose; the system prompt explicitly forbids both, AND the post-processor strips fences anyway. Defense in depth — the prompt asks nicely, the parser cleans up if it ignored.
- **`render_prompt(target, question) -> (system, user)` returns a 2-tuple.** `system` goes to `complete(system=...)`; `user` is the user prompt. Keeps the Claude-side roles aligned: instructions in system, data + question in user. The user's typed question is treated as **trusted** — they're the operator. Brief §7.5's sentinel-fencing applies to *external* untrusted content (alert descriptions, RSS bodies) added by future callers.
- **`_clean_response()` strips at most ONE leading and ONE trailing code fence.** Over-cleaning risks corrupting an otherwise-valid query (a quoted regex like `"[a-z]+"` shouldn't lose characters). Just the obvious ` ``` ` / ` ```language ` / ` ``` ` envelope.
- **Regex validator is a self-check, not a gate.** Returns `ValidationResult(target, query, errors, warnings)`. **Errors** = "this is structurally broken" (unbalanced quotes/parens, YARA-L wrapper for UDM, empty query). **Warnings** = "this might be wrong but I'm not sure" (unrecognised field prefix). The CLI surfaces both but never refuses to run; final judgment is the operator's.
- **UDM-specific: hard-reject `rule { ... }` and YARA-L block keywords.** Brief §6.5 explicitly lists this pitfall — the YARA-L detection-language wrapper is a frequent Claude failure mode for UDM Search. Detected via `re.search(r"\brule\s+\w*\s*\{")` and a multiline regex for `events:` / `match:` / `condition:` block markers at end-of-line.
- **TMV1: known-fields list is the leakage check.** Field names are camelCase and small (~20 known); unknowns warn rather than error because V1 ships new fields fairly often. Known boolean keywords (AND/OR/NOT) explicitly removed from the candidate-field set so they don't trigger a false-positive warning.
- **Quote counter ignores backslash-escaped quotes.** The escape rule is naive (a `\"` is not counted) but matches both syntaxes' simple quoting. A test exercises this case explicitly: `"FOO\"BAR"` is balanced.
- **`generate_query(bridge, target, question) -> GeneratedQuery`** is the one-call entry point: prompt → bridge → clean → validate → return. `GeneratedQuery` carries the raw response too so debug tooling and the upcoming Textual screen can show "what Claude actually said" alongside the cleaned-up query.
- **CLI: `secops-term ai query QUESTION --target udm|tmv1 [--debug-ai]`.** Question is a positional arg; target defaults to `udm` for grade-school ergonomics. `--debug-ai` toggles full-prompt/response capture in the audit log (sensitive — flagged in help text). Composes the bridge via `_build_ai_bridge_candidates()` (factored out as a module-level helper so tests can monkeypatch it cleanly).
- **CLI: `secops-term ai status`** runs `health_check()` against each candidate and prints a table. Useful for "why isn't `ai query` working" — separates "claude binary missing" from "pyperclip can't reach the clipboard" without making the user read source.
- **`_build_ai_bridge_candidates()` order: headless first, clipboard last.** Headless round-trips without operator interaction; clipboard requires manual paste. The composer picks the first healthy one. Phase 4.3 inserts MCP between them.
- **CLI clipboard `response_provider` reads stdin via `asyncio.to_thread(input)`.** Plain `input()` would block the event loop and trip ASYNC250; wrapping in `asyncio.to_thread` keeps the loop alive even though there's no concurrent work. The end-of-paste sentinel is `:end` on its own line — chosen because actual queries never start with `:`.
- **`--debug-ai` flag is per-invocation, not config.** Brief §3.5.11 ties it to the run, not the install. Each `ai query --debug-ai` opts in for that one call; no persistent setting that could be forgotten and silently log every prompt forever.
- **35 new tests** across `test_nlp_prompts.py` (6 tests), `test_nlp_validators.py` (12 tests), `test_nlp_query.py` (8 tests), and `test_cli_ai.py` (9 tests). The CLI tests monkeypatch `_build_ai_bridge_candidates` to a stub that returns a canned UDM query — no real Claude or clipboard interaction, no real subprocess.

## Chunk 4.3 — MCP server (Transport B), NLP screen, doctor extension

- **MCP package split into three layers: `tools.py`, `gate.py`, `server.py`.** `tools.py` is the pure-Python tool surface (no FastMCP, no HTTP) — testable in isolation, importable without optional deps. `gate.py` is the policy layer (auth + rate-limit). `server.py` glues them to FastMCP via `MCPGate.invoke`. Decoupling means the gate logic is exercised by 30+ tests without spinning a network server, and FastMCP can be swapped or upgraded without touching the policy.
- **Loopback-only validation is at IP-address level, not hostname.** `_validate_loopback` parses via `ipaddress.ip_address()` and rejects anything that isn't `is_loopback`. `localhost` is **not** accepted — DNS could be lying. `0.0.0.0` (all-interfaces) is rejected with the same error. Test suite parameterises over a battery of risky bind targets (`8.8.8.8`, `169.254.169.254` AWS metadata, `192.168.1.1`, `::`, `2001:db8::1`).
- **Privileged ports rejected — port must be in 1024..65535.** Defence in depth: even though `_validate_loopback` already gates the bind, refusing privileged ports avoids surprising privilege-escalation prompts on Linux/macOS and stops a misconfigured `--port 80` from clashing with whatever's on 80 already.
- **Bearer-token auth is a per-call `__authorization__` smuggle key.** FastMCP's transport layer doesn't expose request headers consistently across stdio / streamable-http / sse, so we have the gate accept the bearer token through a special argument key the client sets. The key is stripped from the args before audit emission and before reaching the tool handler — token never leaks to logs even on 401 path. Test asserts both directions (success path and unauthorised path) keep `__authorization__` out of the audit entry.
- **Constant-time token compare via `hmac.compare_digest`.** Defeats timing oracles; same pattern as the Argon2id key-derivation flow from Phase 0. Bearer keyword match is case-insensitive (`Bearer` / `BEARER` / `bearer` all accepted).
- **Sliding-window rate limiter, default 60 calls/min/tool.** `RateLimiter` keeps a `deque[float]` per tool and evicts timestamps older than the window on every `.acquire`. Sliding window beats fixed window because it avoids the boundary-burst (100 calls at 11:59:59 then 100 more at 12:00:01 = 200 in two seconds under fixed window). `set_limit(tool, n)` resets the bucket so a higher limit takes effect immediately.
- **`time_source` constructor hook for deterministic tests.** Default is `time.monotonic` (immune to wall-clock skew); tests inject a callable returning a controlled `t` so window-sliding behaviour can be asserted in milliseconds rather than wall-clock waits. `RateLimited.retry_after_s` is computed from the oldest in-window timestamp + window − now, so it shrinks as time passes (test asserts).
- **Five MCP tools: `search_iocs`, `run_retro_hunt`, `summarize_alert`, `nl_to_udm`, `nl_to_v1`.** Direct mapping of brief §7.2's required surface. Each is a `ToolSpec(name, description, input_schema, handler)` with a JSON-Schema `input_schema` so the FastMCP `tools/list` response is correct without per-tool boilerplate. Argument validation lives in the handler for now (re-checking after JSON-Schema is belt-and-suspenders against client-side schema drift).
- **`nl_to_udm` and `nl_to_v1` return the rendered prompt, not an executed query.** The MCP server doesn't call the AI bridge here — Claude is on the other side of the wire. The tool returns `{system_prompt, user_prompt, validator_hint, guardrail, validator_module}` so Claude can prompt itself with the cheat sheet, then optionally call back into a future `validate_query` tool for a structural self-check. Brief §6.5's never-auto-execute rule is restated in the response payload via the `guardrail` field — defence in depth even if Claude ignores the system prompt.
- **`summarize_alert` returns structured fields, not an LLM summary.** The MCP server's job is "expose data"; the LLM client (Claude) does its own summarising. Cuts a Claude → us → Claude → us round-trip and keeps the AI usage decision on the operator side.
- **Tool args validated as `int`, not `int|str`, on `run_retro_hunt`.** A string `"5"` is rejected — JSON-RPC clients should send integers; tolerating strings invites silent bugs later. Test asserts `{"ioc_id": "abc"}` raises `ToolError`.
- **Lazy imports inside handlers.** `_search_iocs` doesn't import the IOC store until called; same for `_summarize_alert` and the alerts ingest. Keeps the `MCP_TOOLS` registry cheap to import (matters for `--help`-style invocations and for `tests/mcp/test_tools.py::test_registry_specs_have_schemas` which doesn't need the DB).
- **`MCPGate.invoke` is the single audit point.** Emits `kind="mcp_tool_call"` with `{tool, args (sans bearer), ok, error?, latency_ms}`. Failures surface the error class and message tail (200 chars max). Best-effort emit (`contextlib.suppress(Exception)`) — same pattern as the AI bridge audit and the HTTP client audit.
- **`secops-term mcp-serve --host 127.0.0.1 --port 8765 [--auth-required] [--rate N]`.** CLI flags map directly to `build_fastmcp_server` constructor args. `--auth-required` reads the bearer token from `secops-term:mcp:default/bearer_token` in the keyring; missing → exits 1 with a clear "set one via the wizard" message. Default port 8765 chosen because it's not a common service port and easy to remember.
- **Doctor extended: `_check_vision_one`, `_check_deep_security`, `_check_ai_bridge`.** The V1 + DS checks are the ones DECISIONS.md claimed in Chunk 3.3 but never actually shipped (caught during Chunk 4.3 wiring — added now). The `ai bridge` check is offline-only: scans for `claude` on PATH and `pyperclip` importable, returns a comma-list of available transports. Subprocess invocation is left for `secops-term ai status` / `config test-all`.
- **`QueryScreen` (Textual, Phase 4.3) replaces the stub.** Same DataTable + detail-Static skeleton style as the other phase-replaced screens, but the layout is `Input` (question) → `Static` (generated query) → `Static` (validation panel). Bindings: `r` generate, `t` toggle target (UDM ↔ TMV1), `c` copy to clipboard, `escape` back. The bridge composer raises if no transport is healthy — the screen surfaces the error in the status line and the user can fix the install (Claude binary missing) without leaving the TUI.
- **Clipboard-fallback in TUI raises `NoTransportAvailable` rather than hanging on stdin.** A real paste-modal would need a multi-line textarea + a "submit" button — out of scope for Phase 4.3. The CLI `secops-term ai query` keeps the operational clipboard flow; the TUI stays pure read-then-react.
- **51 new tests** across `tests/mcp/test_gate.py` (12 tests), `tests/mcp/test_tools.py` (16 tests), `tests/mcp/test_server.py` (15 tests). The wire-protocol layer (FastMCP) is smoke-tested via `build_fastmcp_server` returning a `FastMCP` instance — actually running the HTTP transport in a unit test would be flaky and slow; integration testing is the operator's `secops-term mcp-serve` smoke run with a real Claude Code chat client.
- **`summarize_alert` test mocks `alerts.ingest.ingest_all`** instead of stitching together V1/Chronicle/DS configs. The end-to-end alert ingest has its own coverage (Chunk 3.3, 10 tests); the MCP layer just needs to verify that "given a known alert in the result, the tool finds it by `dedupe_key`". Monkeypatching the ingest gives a clean, fast test.

## Chunk 5.1 — Notifications: generic_json, slack, teams + orchestrator

- **Three concrete notifiers register at module import via `@NOTIFIERS.register("name")`.** Same registry pattern Phase 0 set up; `discover()` walks the package and the autouse `_isolate_plugin_registries` fixture in `conftest.py` snapshots+restores so synthetic test registrations don't leak between tests. The Phase-0 test that asserted "no concretes shipped" was rewritten to assert the Phase-5 reality (the three are present).
- **Channel reference format `{notifier}:{instance}`** is the playbook-facing API. Orchestrator splits on the first colon; both halves must be non-empty. Notifier-name → registered class; instance → keyring service-name segment. Multi-instance is the norm (`slack:soc-alerts` + `slack:escalations`); each shares the registered class but pulls its own keyring entry.
- **Severity → vendor visual (locked in tables):** `info=#36a64f / 107C10` (green), `warn=#ff9933 / FF8C00` (orange), `error=#cc0000 / D13438` (red). Slack uses CSS hex; Teams MessageCard uses 6-char hex without `#`. Locked in module-level dicts so a typo in one place fails one obvious test.
- **`generic_json`: URL precedence `constructor → keyring → error`.** URL in `config.toml` is the default; users can opt-in to keyring storage for the URL itself (when it embeds an auth segment). Optional bearer token always lives in keyring (`bearer_token` field). Two templates — `standard` (full envelope) and `compact` (just summary/severity/body) — selectable per-instance.
- **`slack` and `teams`: webhook URLs MUST live in keyring.** Brief §6.6 treats them as credentials. `from_config` actively rejects `url` / `webhook_url` keys in the TOML block — silently ignoring them would let a typo land a webhook URL on disk. Keyring service: `secops-term:notifications.<notifier>:<instance>` / `webhook_url`.
- **Slack-specific success rule: 200 + body `"ok"`.** Slack returns 200 with `invalid_payload` body on malformed payloads; treating any 2xx as success would silently swallow those. Same pattern for Teams: 200 + body `"1"` is success; everything else is `delivered=False` with the response text in `detail`.
- **`Notifier.send()` never raises on transport failure** — that's part of the Protocol contract. Returns `NotifyResult(delivered, detail, latency_ms)`. The playbook engine treats `delivered=False` as a *retryable* error (most are transient: 5xx, brief outages). `NotifierError` for build-time issues (missing webhook URL) flows out of `from_config` / `send` only when the failure is structural.
- **Health probes are real low-noise sends.** Slack/Teams don't expose an auth-validating endpoint that doesn't post to the channel, so we send a `(automated; safe to ignore)` probe message. Generic JSON probes the same payload shape used for real sends so mis-shaped payloads fail probe rather than waiting for the first real notify. Cost: every `config test` adds one message to the channel — operators are expected to silence the bot or use a `secops-term-probes` sub-channel.
- **`MessageCard` (legacy Teams format) chosen over Adaptive Cards.** Microsoft has been rolling out Adaptive Cards as the replacement, but MessageCard works on every Teams tenant we care about and avoids tenant-specific schema bugs. Documented in the module docstring with "one-spot edit when the upstream cutover lands". The fact that this is locked in 2026 will look anachronistic in 2027 — leaving a clear trail.
- **Orchestrator `build_health_targets` returns `(notifier, channel_label)` pairs** to mirror `intel.orchestrator.build_health_targets`'s shape. Doctor and `config test-all` can mix providers and notifiers in a single concurrent probe without tracking two different return types.
- **Health-target builder skips notifiers it can't construct.** A missing webhook URL in the keyring raises `NotifierError` from `from_config` / a `_get_webhook` call; the orchestrator catches and skips so doctor stays offline-friendly. The error surfaces clearly via `config test <channel>` instead.
- **54 new tests** across `test_generic_json.py` (16), `test_slack.py` (12), `test_teams.py` (10), `test_orchestrator.py` (12 incl. parametrised malformed-channel cases), plus the Phase-0 registry test rewritten. All HTTP exchanges mocked via `respx`; keyring via `_fake_keyring` modules injected into `SecretsConfig.keyring_module` (same fixture pattern as the chronicle / abuse_ch suites).

## Chunk 5.2 — Playbook engine: Pydantic schema, sandbox templating, retry, audit

- **Schema split: discriminated union for both `Trigger` and `Step`.** `Trigger = Annotated[IocAddedTrigger | ManualTrigger | ScheduledTrigger, Field(discriminator="type")]`. Pydantic v2 routes to the right concrete class by the `type` literal; unknown values raise `ValidationError`. Same pattern for `Step` across `api_call` / `retro_hunt` / `notify` / `summarize`. Adding a new variant is a one-spot edit and mypy + Pydantic both enforce exhaustive handling downstream.
- **`extra="forbid"` on every model.** A typo'd field in YAML (`channnel: slack:x` instead of `channel:`) is rejected at load time, not silently ignored. Same posture as the Phase-3 alerts schema; the operator sees the validation error before a runtime mid-execution failure.
- **`when:` accepts both `{{ expr }}` and bare `expr`** — the engine strips a single envelope before evaluating. Brief §6.4 examples wrap; bare is cleaner for short conditions. Internal note: we use `sandbox.evaluate` (single expression, value returned) for `when:`, not `sandbox.evaluate_template` (template, returns string) — truthiness is what we care about.
- **Step ids are identifier-shaped (letters, digits, underscores).** Field validator on `_BaseStep.id` rejects anything else — avoids quoting surprises in `{{ steps.<id>.* }}` references downstream. Duplicate ids within a playbook → ValidationError with the duplicates listed.
- **Hard playbook timeout default 5 min, configurable up to 30 min.** Brief §6.4: 5 min default. We cap at 1800s (30 min) — a runaway `summarize` step waiting on the AI bridge shouldn't be able to monopolise the runner indefinitely. Timeout fires via `asyncio.wait_for` and surfaces as a synthetic `__timeout__` step in `PlaybookRun.steps` so the table row shows the failure cleanly.
- **Engine is decoupled from concrete clients via `runners: Mapping[str, StepRunner]`.** Tests inject stubs; production uses `build_default_runners()` (or `build_runners_with_ioc()` for trigger-bound execution). Lets the engine ship without a circular dependency through the chronicle / V1 / DS / notifications packages, and keeps the unit tests fast.
- **Retry budget: 3 attempts, exponential backoff (0.2s → 0.4s → 0.8s).** Brief §6.4: "Per-step retry: 3 attempts, exponential backoff, retryable HTTP codes only." Distinction made via two error classes: `StepError` (non-retryable, single shot) vs `RetryableStepError` (retried up to budget). Runners pick which to raise based on the underlying failure shape — `notify` treats `delivered=False` as retryable; `api_call` always raises `StepError` until Phase 6 ships the allow_write gating.
- **Engine flags AI step output untrusted via `isinstance(step, SummarizeStep)`.** Brief §7.5 rule 1 (AI output may NOT gate control flow) is operationalised by tracking a `untrusted_ids: set[str]` of step ids that produced AI-derived output. The `_sanitize_for_when` helper strips those entries from the `steps` dict before passing it to `when:` evaluation — the AI step's output is invisible to subsequent gating, but visible to subsequent `notify` text rendering (display-only, brief §7.5 rule 1 explicitly allows this).
- **Sandbox `when:` violations fail the step rather than the whole playbook.** A malicious / malformed `when:` expression (e.g. `__import__('os')`) is caught as `SandboxError`, the step is recorded as failed with a `when:` prefix, and the playbook continues to subsequent steps. No information leak: the audit entry includes the exception class name but not the source expression, which is already in the YAML on disk anyway.
- **`dry_run=True` walks the same control flow.** Renders every templated string, evaluates every `when:` expression, but never invokes the runner. Each step's `output` becomes `{"dry_run": True, "rendered": <params>}` so the operator can inspect what *would* go out. Brief §6.4 explicitly requires this; it's also what the Textual screen uses for in-TUI inspection.
- **`audit_logger.emit` per step with `kind="playbook_step"`** including `playbook`, `step_id`, `step_type`, `skipped`, `ok`, `dry_run`, `error`, `latency_ms`, `attempts`. Same `contextlib.suppress(Exception)` posture as the AI bridge audit — a logger blow-up never breaks the step path. Playbook timeout produces a synthetic `__timeout__` audit-less marker (the timeout itself is logged as the step that was running).
- **Loader uses `ruamel.yaml` (already a project dep) with `typ="safe"`.** Rejects unknown tags and arbitrary Python objects. Was tempted to bring in `pyyaml` but `ruamel.yaml` was already declared in `pyproject.toml`. Path-traversal attacks on `load_playbook_by_name` go through `paths.safe_join`; the test asserts a `../../etc/passwd` lookup raises `TraversalError`, not a `FileNotFoundError`.
- **50 new tests** — `test_schema.py` (17, every Pydantic invariant), `test_loader.py` (12, parse/path safety), `test_engine.py` (21, when/retry/audit/timeout/AI-untrusted gating). The "AI output hidden from when:" test is the brief-§7.5-rule-1 enforcement assertion: after a `summarize` step lands `text` in `steps.ai`, a subsequent `when: '{{ steps.ai.text == "x" }}'` blows up with `NotFound` because the sanitised context the engine hands `when:` evaluation has stripped the `ai` entry. The same ai output flows fine into a downstream `notify` template — verified by a parallel test.

## Chunk 5.3 — Example playbooks, CLI, Playbooks screen, doctor extension

- **Three example YAMLs ship in `secops_term/playbooks/examples/`** — `high-conf-ioc-followup.yaml`, `daily-feed-pull.yaml`, `weekly-osint-roundup.yaml`. Per brief §6.4. `secops-term playbooks init` copies them to `~/.secops-term/playbooks/`; `--force` overwrites existing files (otherwise skipped with a clear message).
- **`high-conf-ioc-followup` uses `ioc_added` trigger + `confidence_gte: 80` filter.** Two parallel-style retro-hunt steps (Chronicle + V1) followed by a notify gated on `steps.retro_chronicle.status == 'queued' or steps.retro_v1.status == 'queued'`. Comment in the YAML is explicit that the "hits" referenced are queue-side acknowledgement (Phase 5 surface), not real hit counts (Phase 6 worker landing).
- **`daily-feed-pull` is a `scheduled`-triggered notify-only stub.** Phase 5 ships `intel pull` as a CLI command, not a playbook step (brief §13: api_call gating is locked read-only). The playbook is a heads-up to the SOC channel; a comment in the YAML directs the operator to wire it into a cron entry until Phase 6 ships the scheduled-trigger worker.
- **`weekly-osint-roundup` uses the `manual` trigger.** Notify-only digest. Comment notes that Phase 6 may add a `summarize` step asking Claude to render a richer roundup, with the brief-§7.5 reminder that AI output is display-only.
- **CLI: `secops-term playbooks list/show/init/run`.** `list` prints a table with name/trigger/step-count/description. `show NAME` prints the parsed playbook (post-validation) so operators can inspect what the engine sees vs. what the YAML says. `init` copies bundled examples; `run NAME [--dry-run] [--ioc-id N]` executes. `--ioc-id` is required for `ioc_added` triggers — otherwise the engine has no IOC context for retro-hunt enqueue or template interpolation.
- **`run` exits 1 on any step failure**, 0 only when every non-skipped step succeeded. Mirrors the unix-tools convention — a CI/cron caller can `&&`-chain on success. The detailed step-by-step table prints regardless so operators see what happened.
- **`run` uses `build_runners_with_ioc(ioc_id)` when an IOC is supplied,** else `build_default_runners()`. The retro_hunt runner needs an IOC binding to know what to enqueue; without one it raises `StepError("retro_hunt runner needs an IOC binding")` so a manually-triggered playbook with a retro-hunt step fails fast rather than enqueueing a job for IOC `None`.
- **`PlaybooksScreen` (Textual) is a list + dry-run view.** Real execution (with side effects) is intentionally CLI-only so the operator-confirmation flow stays explicit. `r` refreshes the list, `d` dry-runs the highlighted playbook, `escape` returns. Same DataTable + detail-Static skeleton as the other Phase-replaced screens.
- **Doctor extended: `_check_notifications` and `_check_playbooks`.** Both offline-friendly. Notifications check counts configured channels; playbooks check counts loadable / unloadable YAMLs. Live notifier health probes still live behind `config test-all`. Playbooks with `PlaybookError` show `<count> found, <bad> invalid: <first-error>` so operators see at-a-glance which file is broken.
- **`Playbook.steps` rejects empty list** (`min_length=1`) — zero-step playbooks have no value and are almost always operator error. `max_length=50` is a sanity ceiling (the brief doesn't specify; 50 covers every realistic SOAR-light workflow).
- **107 new tests in Phase 5 total** (54 notifications + 50 playbooks + 13 CLI playbook tests). Test ratio holds: every step type, every trigger type, every retry / timeout / audit edge has at least one test. The CLI tests assert the exit-code contract (1 on failure, 0 on success-or-dry-run-success).


## Chunk 6.1 — VirusTotal + GreyNoise intel providers

- **VirusTotal uses the Intelligence Search API for `pull()`** — `GET /api/v3/intelligence/search?query=type:malware+p:5+&limit=40&order=last_submission_date-`. Community (non-Intelligence) keys receive HTTP 403 from this endpoint; the provider degrades gracefully to an empty list rather than raising. Brief §5 decision: "VT probe uses `/users/<owner>` to avoid burning the daily quota." The `owner` field in config is the VT username; if absent, `health_check()` returns `ok=False` immediately with a clear config-guidance message.
- **VirusTotal emits three IntelRecord rows per file object** (sha256, sha1, md5) — one per hash type present in the attributes. Store dedup on `UNIQUE(type, value)` handles the case where the same hash arrives from OTX and VT independently.
- **VirusTotal `limit` is clamped to [1, 300]**. The NVD cap is 2000, the AbuseIPDB cap is 10000. VT Intelligence Search has an undocumented practical limit around 300 per page. Documented in the class.
- **GreyNoise uses GNQL** (`GET /v2/experimental/gnql?query=classification:malicious&size=100`). Enterprise and Research account holders get full GNQL access; Community keys get 401/403 from this endpoint and the provider degrades gracefully to empty list. Health check uses `GET /ping` which is auth-validating and zero-quota. GN ping response includes `offering` key (enterprise/community/etc) surfaced in the health detail.
- **GN `last_seen` format can be `"YYYY-MM-DD"` or ISO-8601T string** — both handled in the `since` filter. The ISO variant has a trailing `Z` which Python `fromisoformat` doesn't accept directly in 3.10 but handles fine after `.replace("Z", "+00:00")`.
- **31 new tests** — VT (15) and GN (16). VT tests cover hash mapping, auth header, 403 degradation, `since` timestamp filtering, owner-less health probe, and quota detail parsing. GN tests cover GNQL params, key header, 401 degradation, `since` date filtering, context construction, and offering tier.

## Chunk 6.2 — AbuseIPDB + NVD intel providers

- **AbuseIPDB pulls the IP blacklist** (`GET /api/v2/blacklist?confidenceMinimum=75&limit=1000`). Response includes `lastReportedAt` per entry enabling `since` filtering post-fetch (the API has no server-side date filter on the blacklist endpoint). `confidence_minimum` clamped to [25, 100]; values below 25 are not supported by AbuseIPDB. `abuseConfidenceScore` preserved as `IntelRecord.confidence`.
- **AbuseIPDB health probe targets `8.8.8.8`** (`GET /api/v2/check?ipAddress=8.8.8.8`) — Google DNS, consistently reports a confidence score near zero, and probing it validates auth with negligible quota cost.
- **NVD is a public government API** — `api_key` is optional. Without it, NVD enforces aggressive rate limiting (5 req/30 s vs 50 req/30 s with key). The provider's `_get_api_key_optional()` returns `None` rather than raising when no key is configured, and `pull()` + `health_check()` succeed either way. This is the only intel provider that doesn't require authentication as a precondition.
- **NVD uses `pubStartDate` + `pubEndDate` query params** for date-windowed pulls (ISO format `YYYY-MM-DDTHH:MM:SS.000`). When `since` is supplied it sets `pubStartDate`; otherwise the window is `now - days_back` days. `min_cvss_v3=7.0` (configurable) drops CVEs below the threshold via post-processing — NVD doesn't support a score filter query param.
- **NVD emits `cve` type IntelRecords** with `source_ref = "https://nvd.nist.gov/vuln/detail/{CVE-ID}"`. CVSS v3 severity becomes a tag (`"critical"`, `"high"`, etc. lowercased). English description populates `context` (truncated to 200 chars). CVSS version preference: v3.1 > v3.0 > none.
- **30 new tests** — AbuseIPDB (15) and NVD (15). AbuseIPDB tests cover confidence clamping, key header, `since` post-filtering, ISP+usage context, health probe detail. NVD tests cover unauthenticated pull, optional key header, CVSS filter, `since` param, severity tag mapping, and auth-mode health detail.

## Chunk 6.3 — STIX 2.1 export

- **STIX 2.1 SCO IDs are deterministic (UUIDv5)** using namespace `00abedb4-aa42-466c-9c01-fed23315a9b7` (the STIX 2.1 spec SCO namespace, §2.9) seeded with `"{ioc_type}:{value}"`. The same IOC always produces the same STIX ID across exports, enabling consumers to merge overlapping bundles without creating duplicates.
- **Bundle ID uses a fresh random UUIDv4** on each export. Bundles are ad-hoc snapshots; deterministic bundle IDs would imply equivalence between two exports that differ only in which IOCs were filtered, which is semantically wrong.
- **Duplicate STIX IDs collapsed to first occurrence.** If the same IOC value appears twice in the input (e.g., same sha256 arriving from two providers), the bundle contains one STIX object. The `seen_ids` set handles this without needing to pre-sort.
- **CVE → `vulnerability` SDO** with `created`/`modified` from `IOC.first_seen`/`last_seen`, `external_references` pointing to `nvd.nist.gov/vuln/detail/{id}`. File hash IOCs produce `file` SCOs with a single `hashes` key rather than trying to merge sha256+sha1+md5 from separate store rows — consumers that want a merged file object can correlate by hash value.
- **CLI `intel export --format stix`** writes to stdout (default) or `--out FILE`. `--type` and `--limit` mirror `intel list` so operators can export a subset. Unknown `--format` exits non-zero immediately with a clear error message. Lazy import of `stix_export` inside the command keeps the import footprint of short-lived CLI invocations minimal.
- **`_stix_timestamp` emits millisecond-precision UTC strings** (`YYYY-MM-DDTHH:MM:SS.<ms>Z`) per STIX 2.1 §2.1 timestamp spec. Python `datetime.microsecond // 1000` gives the millisecond component.
- **24 new tests** — STIX unit tests (19, covering all 9 IOC types, deterministic ID, bundle dedup, and timestamp format) + CLI export tests (5, covering empty store, type filter, file output, and unknown format rejection).
- **85 new tests total in Phase 6.1–6.3.** Running count: 968 passed / 6 skipped (unchanged skips: 4 POSIX path-traversal + 1 POSIX mode + 1 orchestrator covered by CLI end-to-end).

## Chunk 6.4 — PyInstaller builds + README + version tag

- **Version bumped from `0.1.0` to `0.6.0`** in `pyproject.toml`. The version is read at runtime via `importlib.metadata.version("secops-term")` so the CLI and TUI header auto-reflect the bump without any code change.
- **PyInstaller spec uses directory mode (not onefile).** `--onefile` is convenient but produces single-file executables that decompress to a temp directory on every launch, which triggers AV heuristics on security workstations. Directory mode keeps the bundle transparent and avoids the temp-dir extraction. The entry point executable is `dist/secops-term/secops-term[.exe]`.
- **Hidden imports declared explicitly in `secops_term.spec`.** The plugin registries (`PROVIDERS.register`, `NOTIFIERS.register`) call `importlib.import_module` dynamically. PyInstaller's static analysis misses these. All 7 provider modules and all 3 notifier modules are listed in `hiddenimports`. Keyring backends for all three platforms are also listed so a build from any host includes the backends for all targets (the OS-specific ones are no-ops on non-native builds but don't bloat significantly).
- **Textual collected via `collect_all("textual")`** in the spec. Textual bundles CSS and widget templates inside the package; PyInstaller needs `collect_all` rather than bare hidden imports to pick up the data files. `_textual_datas`, `_textual_binaries`, and `_textual_hiddenimports` are merged into the `Analysis` object.
- **`scripts/build_dist.py` is the recommended build entry point.** It locates PyInstaller via `python -m PyInstaller` (prefers the venv copy) with a fallback to `shutil.which("pyinstaller")`, validates output, and prints platform-specific post-build instructions (signtool on Windows, codesign on macOS, tarball/AppImage on Linux).
- **`[build]` optional extra already existed in `pyproject.toml`** declaring `pyinstaller>=6.7`. No new dependency entries needed.
- **README updated** to reflect v0.6.0 status: full phase summary table, complete CLI reference, intel provider table, STIX export usage examples, and distributable builds section. Status line changed from "Phase 0" to "v0.6.0 — all six phases complete."
- **0 new tests** for this chunk — the spec and build script are build-time artifacts, not testable at import time. PyInstaller builds are exercised manually per the README before tagging.
